# subgroup-web — design

Architecture notes for contributors: how the library is layered, the backend
seam, the WebGPU exactness band and its derivation, the memory model, and
where this implementation deliberately departs from pysubgroup 0.9.0's
architecture. Behavior (what results mean) is pinned in [spec.md](spec.md);
this document explains how it is computed fast without giving up exactness.
Measured numbers below come from the committed runners
(`scripts/bench.mjs --gates`, `scripts/measure-a12.mjs`, the Playwright band
gate) on the reference machine (Apple Silicon, 18 cores, Metal 3,
Chromium 149); BENCHMARKS.md is regenerated by every `pnpm gate`.

## 1. Layering

```
table/  desc/  targets/           data model, selector spaces, QFs (spec §§1–6)
bitset/                           selector atlas + SWAR AND/popcount kernels
search/                           algorithms, central scorer, TopK (spec §7)
backends/ cpu|webgpu              BatchEvaluator implementations
results/                          stat tables, filters, serialization (spec §8)
```

Search algorithms (`apriori`, `dfs`, `dfsNumeric`, `bestFirst`, `beamSearch`,
`patternTree`, `generalizingBFS`, `exhaustive`) never touch rows, covers, or
floats from a backend other than through one seam:

## 2. The BatchEvaluator seam

`BatchEvaluator` (src/backends/types.ts) is the only backend abstraction.
It answers two batch shapes — flattened ascending selector-id tuples
(apriori levels, worker shards, GPU dispatches) and one-parent-many-extensions
(dfs / best-first / beam expansion, `op: "and" | "or"`, the latter serving
generalizingBFS's disjunction space) — and returns **statistics only**
(`StatsBatch`: size, positives, sum, excessSum, …, struct-of-arrays,
transferable). Quality and optimistic-estimate computation happens centrally
in f64 (src/search/scorer.ts) through the task's QF.

Consequences:

- **Bit-identical ranking across backends.** Backends may differ in how they
  count; they cannot differ in how counts become qualities. The
  backend-identity gates (Node workers, browser, WebGPU) assert equality of
  `(canonicalKey, quality-bits)` sequences, not approximate agreement.
- **Exactness burden is localized.** A backend is correct iff its statistics
  are correct (integers) or conservatively banded (GPU f32 sums, §4).
- **Workers and GPU are schedulers, not semantics.** The worker pool shards
  tuple batches and stitches `StatsBatch`es back in candidate order; the GPU
  evaluator returns one `StatsBatch` per dispatch group. Neither ranks.

**Backend resolution** (src/search/engine.ts `resolveEvaluator`): explicit
requests are binding — `backend: "webgpu"` throws when no factory is
registered or the GPU fails (it falls back to CPU, with a note, only for
the documented applicability declines), and an explicit `workers` option
propagates pool spawn failures. `backend: "auto"` is a degradation ladder:
WebGPU when a factory is registered, the task is heavy (rows × selectors ≥
2²⁴) and applicable → CPU worker pool (heavy tasks; skipped under
`workers: false`, the single-thread spelling) → single-thread CPU. Every
'auto' tier failure is caught and its reason accumulated into
`results.backend.note` (joined with "; ") — an 'auto' search never dies on
a missing/broken accelerator, and the note says what degraded and why.

## 3. CPU backend

The atlas (src/bitset/atlas.ts) is one selector-major `Uint32Array` — 32
rows/word, `wordsPerRow = ⌈nRows/32⌉` — built once per task (lazily: the GPU
codes-mode path never builds it, §5). Covers are word-wise ANDs into reusable
scratch; conjunction statistics stream out of fused AND+popcount (SWAR)
kernels without materializing the cover (`fusedBinaryCounts`,
`numericStatsFromCombinedBits`): the final level of every tuple batch is
computed as `stats(prefix ∧ row)` directly.

Tuple batches exploit apriori's parent-major candidate order: one scratch row
per depth, re-derived only for the suffix that differs from the previous
tuple (longest-shared-prefix; §7 measures why this beats a retained cache).
Extension batches take the parent cover explicitly — dfs/best-first/beam hold
their O(path-depth) covers themselves, which is the only cover state those
algorithms need.

**Worker pool** (src/backends/cpu/pool.ts): `node:worker_threads` in Node,
`Worker` in browsers, same wire protocol (src/backends/cpu/protocol.ts).
Two data regimes, chosen at init: SharedArrayBuffer (Node always; browsers
when `crossOriginIsolated`) shares the atlas zero-copy; otherwise the atlas
ships once per worker as a transferred copy (the GitHub Pages regime — no
COOP/COEP headers — is the default-tested one). Batches below
`localThreshold` (default 256 candidates) run inline: sharding 4 candidates
across 8 workers costs more in messaging than the AND-chains. Workers are
process-cached keyed by script+regime (`terminateCachedWorkers()` to drop);
spawn is ~18 ms warm. Init/reply ids are globally unique and echoed so a
reused worker can never satisfy a stale request. Shard results are stitched
into one `StatsBatch` in submission order, so downstream scoring is
order-identical to single-thread — this is what the m6-backend-identity gate
checks (bit-identical across 1 thread / SAB pool / copy pool).

## 4. WebGPU backend and the §12 exactness band

The GPU evaluator (src/backends/webgpu/evaluator.ts) mirrors the CPU one at
the same seam. Applicability is static per task (`gpuApplicable`): binary and
frequent-itemset targets always; numeric targets only for sum-family plans
(no std/median/order/tail); EMM never — those run CPU even when
`backend: "webgpu"` is requested (recorded in `SubgroupResults.backend`).

Kernels (src/backends/webgpu/wgsl.ts):

- `countsKernel` / `numericKernel`: one workgroup (256 threads) per
  candidate; each thread strides the cover words, AND-chaining atlas rows
  in-register (`coverWord` recomputes the chain — no cover is ever stored),
  popcounts or gathers target values, then a workgroup tree-reduction.
- `countsPairsKernel`: the arity-2 apriori level is the P2 hot path
  (32 640 pairs × 2M rows); prefix and target tiles are staged through
  workgroup shared memory (1536-word tiles) and reduced with u32 `atomicAdd`
  — integer, order-free, exact.
- `atlasBuildKernel`: builds the atlas on-device in codes mode (§5).

Dispatches are grouped under `maxWordsPerDispatch` and the device's binding
limits (atlas chunks across bind slots); all groups submit, then **one**
staging copy + `mapAsync` per evaluate call (`candBase` offsets each group's
writes into the shared out buffer). `preferredBatchSize = 32768` lets the
engine hand the GPU whole levels — one host sync for P2's level 2. Device
loss surfaces as a typed `BackendError`; per-submit work is bounded by the
chunk/dispatch limits (BRIEF §22-A14).

**Exactness band (BRIEF §12, §22-A7).** Binary/FI statistics are u32 —
integer-exact, no band. Numeric sums are f32 **screening** values with a
per-candidate conservative error bound returned in `StatsBatch.screening`:

- Each thread accumulates ≤ `T₀ = ⌈wordsPerRow/256⌉·32` values sequentially;
  the tree reduction and readback add ~10 more rounding steps, so the
  per-candidate rounding-step count is `T = min(T₀, size) + 10`.
- Standard sequential-summation analysis: `|fl(Σx) − Σx| ≤ T·u·Σ|x|` to first
  order, `u = 2⁻²⁴` (f32 unit roundoff). The kernel accumulates `Σ|x|`
  (absSum) alongside, so the bound is *per candidate*, not global:
  `sumEps = S·u·T·absSum` with safety factor `S = 4` absorbing the
  first-order approximation, the f32 quantization of the uploaded values,
  and absSum itself being an f32 accumulation.
- `excessSum = Σ max(dir·(x−c0), 0)` adds a boundary term: a value within
  rounding distance of the centroid `c0` can land on the wrong side of the
  max. Each term's magnitude is ≤ `|x|+|c0|`, so
  `excessEps = S·u·(T·(absSum + size·|c0|) + size·(max|x| + |c0|))` — the
  first part is accumulation error, the second the misclassification mass.
  `c0` is `Math.fround`ed on the CPU side so both backends test against the
  identical f32 centroid (no representation divergence).

The band is consumed centrally, never inside the backend:

- The scorer shifts screened stats to **upper bounds** (`sum + dir·sumEps`,
  `excessSum + excessEps`) — sound because GPU numeric applicability is
  restricted to plans monotone in these statistics.
- `TopK.couldAdmit(q̄)` is monotone in quality, so an upper bound that fails
  cannot exclude a candidate that exact scoring would admit. When q̄ *could*
  admit, the engine re-scores that single candidate on CPU f64
  (`SearchRun.rescoreExact`: cover from the atlas + `evaluateCover` + the
  same scorer) before it touches the result set. **No screening value ever
  makes a final decision.**
- Pruning compares upper-bound optimistic estimates against the threshold —
  conservative, so no exact-survivor is ever pruned; the extra survivors are
  evaluated and rejected exactly, which cannot change the top-k.

Empirical validation (Playwright gate `m6-gpu-band`, real Metal adapter,
run per gate): every candidate's true f64 statistic lies inside the band —
e.g. credit-g stdnum sweep: 3081 candidates, 0 violations, max observed
`|gpuSum − sum|` = 0 at eps 3.4e-1; planted-numeric-500: 120 candidates,
max err 1.0e-5 vs eps 2.5e-3 (bound ≈ 240× the observed error — conservative
but tight enough that P3 re-scores only a handful of admissions). The
m6-gpu-pruning-chunking gate additionally forces tiny atlas chunks (4 KiB)
and multi-group dispatches (4096-word groups) and asserts oracle equality,
covering the grouping/offset arithmetic.

## 5. GPU codes mode

Uploading a 2M×256-selector atlas costs 64 MB of bandwidth *after* 64 MB of
CPU SWAR work to build it. When every selector is an equality over a
categorical column with ≤ 254 categories (the P2 shape, and the common
demo shape), the evaluator instead uploads the dictionary **codes** — u8,
4 rows per u32 word, packed straight into the codes buffer's
`mappedAtCreation` range so the pack *is* the upload (no intermediate copy;
−1/NA maps branchlessly to the 0xff sentinel, which satisfies no selector;
a selector value absent from the column maps to `categories.length`, an
empty cover) — and `atlasBuildKernel` expands codes to atlas bits on-device
(tail rows masked by `nRows`). `PreparedTask.atlas` is a lazy getter, so the
CPU atlas is never built on this path; end-to-end GPU P2 (prepare + create +
search) is ~93 ms, of which the codes path cut ~30 ms of pack/staging and
~220 ms of CPU atlas build.

## 6. Memory model

Per BRIEF §9, sized for the P2/P4 task (2M rows × 256 selectors,
`wordsPerRow` = 62 500):

| allocation | bytes | lifetime |
|---|---|---|
| selector atlas (CPU or GPU) | nSel · wordsPerRow · 4 = **64 MB** | task |
| prefix scratch | depth · wordsPerRow · 4 = 750 KB (d3) | evaluator |
| StatsBatch buffers | ≤ batchSize · ~12 B ≈ 48 KB–1.5 MB | batch |
| TopK | k · (tuple + aux) ≈ KBs | task |
| GPU codes buffer (codes mode) | nAttrs · rows = 64 MB | create() only |

There is deliberately **no per-candidate cover storage**: covers exist as
one scratch row per depth (CPU), in-register (GPU), or O(path) rows (dfs
family). The P4 gate measures the ledger at run time: atlas = 64 MB =
98.8 % of search allocations, batch = 48 KB, peak RSS 1.21 GB (dominated by
the 2M-row table itself + V8 heap), and fails if the atlas stops dominating
or drifts from ≈64 MB — the canary against accidentally materializing
covers. Results are cover-lazy too: top-k rows carry `(size, positives)`
admission stats (`TopKItem.aux`), so building a result table never re-touches
the atlas unless covers/stat-tables are requested.

## 7. A12 — layer covers: recompute vs cache (measured)

Apriori's level L needs, for every candidate, the cover of its (L−1)-prefix.
The reference materializes per-subgroup representations; BRIEF §22-A12 asks:
re-derive AND-chains from the atlas, or cache parent covers under a byte
budget? `scripts/measure-a12.mjs` (committed; `node scripts/measure-a12.mjs`
after a build) measures three strategies on an apriori-shaped parent-major
arity-3 workload, using the library's own kernels, medians of 5:

| dataset | rows | words/row | parents | candidates | A naive (ms) | B reuse (ms) | C cache (ms) | B retained | C retained |
|---|---|---|---|---|---|---|---|---|---|
| adult d3 level | 48 842 | 1527 | 276 | 31 832 | 76.4 | 44.6 | 47.5 | 12 KB | 1.7 MB |
| synth-2M d3 level | 2 000 000 | 62 500 | 48 | 12 000 | 1203.4 | 736.0 | 747.2 | 488 KB | 12.3 MB |

- **A — naive recompute** (full AND-chain per candidate) pays (L−2) extra
  word-sweeps per candidate: ~1.6× slower. This is what order-oblivious
  evaluation would cost.
- **B — consecutive-prefix reuse** (what `CpuEvaluator.evaluateTuples`
  implements): because apriori emits candidates parent-major, the prefix row
  is re-derived once per *parent*, amortizing to ~1/extensions ANDs per
  candidate. Retains one row per depth.
- **C — retained parent-cover cache** (budget = ∞) does the same ANDs as B
  once each and keeps every cover: it is B plus memory plus lookup — 47.5 vs
  44.6 ms and 6 % slower on the 2M task, while retaining 140×–25× more
  memory (unbounded in nSel²; a *frequent level 2* on the P2 task would be
  32 640 × 250 KB ≈ 8 GB).

**Decision.** A retained cache is dominated at depth ≤ 3: parent-major order
means a cache can never serve a hit that scratch reuse doesn't already serve,
so its best case is tying B while its memory is unbounded without a budget —
and any budget's spill path *is* strategy B. The engine therefore chooses
per candidate, adaptively by tuple order (longest shared prefix with the
previous candidate), which is cache-optimal for sorted input and degrades to
A (still cheap, bounded above by the table) for arbitrary order; dfs-family
algorithms keep their O(path) covers explicitly via `evaluateExtensions`;
the GPU recomputes chains in-register where covers are free. If depth ≫ 3
search ever lands on the roadmap, the measurement above is the template to
revisit the budgeted cache.

## 8. Departures from the reference architecture

Behavioral divergences are adjudicated in COMPATIBILITY.md (ADJ-001…012 with
committed repros); these are the *architectural* redesigns:

- **Vertical atlas vs per-subgroup representations.** pysubgroup wraps each
  subgroup in `BitSetRepresentation` objects holding full-length arrays;
  covers here are transient scratch over one shared atlas (§6, §7).
- **Statistics-only backends.** The reference's QFs compute qualities inside
  representation-specific `calculate_statistics`; here backends return
  integers/sums and one f64 scorer ranks (§2) — this is what makes worker
  and GPU identity *bit*-exact rather than approximate.
- **Central canonical TopK** (spec §3.3): the reference's `heapq` on
  `(quality, subgroup)` tuples makes tie order an artifact of Python object
  comparison (ADJ/MAP-001); here ties break on the canonical description
  order, on every backend, with `couldAdmit` doubling as the band's
  admission test (§4).
- **patternTree vs GpGrowth**: FP-growth-style mining with mergeable integer
  target statistics, restricted to binary/FI where the merge algebra is
  exact (the reference flags its GpGrowth experimental; divergences
  adjudicated, spec §7.10).
- **generalizingBFS vs the reference's generalization fudge**: exact search
  over the disjunction space via OR-covers (spec §7.11) instead of the
  reference's non-admissible quality patch (repro'd in ADJ records).
- **dfsNumeric**: restricted to the QF family its prefix-scoring is provably
  valid for (standardNumeric), with typed errors instead of silently wrong
  results (BRIEF §22-A5).
- **Workers/GPU exist below the semantics line.** pysubgroup has no
  parallelism; subgroup-web's is unobservable in results by construction —
  the m6 identity gates are the proof obligation, and P1–P5 (BENCHMARKS.md)
  show it buys 17.9× over the reference on P1 and 6.6× GPU-over-CPU-pool on
  P2 without touching a single ranked bit.
